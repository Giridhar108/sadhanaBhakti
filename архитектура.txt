Да, для 20 000 пользователей я бы не начинал с микросервисов. Лучший путь: React/Vite на фронте + модульный монолит на бэкенде, но сразу сделать так, чтобы потом можно было масштабировать без переписывания всего проекта.

Важно: 20 000 зарегистрированных пользователей и 20 000 одновременных онлайн-пользователей — это два разных зверя. Для первого хватит довольно простой архитектуры. Для второго уже нужны балансировка, кэширование, очереди, мониторинг и аккуратная работа с БД.

Работа с API

Я бы сделал единый слой API:

shared/api/
  httpClient.ts
  endpoints.ts
  types.ts

Примерно так:

export const httpClient = fetch или axios instance

А внутри фич:

features/start-japa/api/startJapaSession.ts
features/start-japa/model/useStartJapaSession.ts

Для серверного состояния лучше использовать TanStack Query: он решает кэширование, повторные запросы, loading/error-состояния и инвалидирование данных. Zustand оставить для локального состояния интерфейса: открытое меню, тема, модалки, временные UI-настройки.

Оптимизация фронтенда

На уровне роутов нужно делать lazy-loading:

const DashboardPage = lazy(() => import('@/pages/dashboard'));

React официально поддерживает lazy вместе с Suspense, чтобы загружать части интерфейса по мере необходимости. ()

Для 20 000 пользователей важно:

разделение bundle по страницам

ленивая загрузка тяжёлых экранов
оптимизация изображений
WebP/AVIF
не хранить лишнее в global store
не делать огромные таблицы без виртуализации

Главная идея

Сначала делаем модульный монолит:

один backend-проект
одна база данных
чёткое разделение по модулям
чистые границы между доменами

Не надо сразу делать 10 микросервисов. Это преждевременная боль: деплой сложнее, отладка сложнее, DevOps сложнее, а пользы на старте мало.

Подключения к БД

Когда backend начнёт масштабироваться в несколько инстансов, важно не убить PostgreSQL количеством соединений. Для этого используют connection pooling. Например, PgBouncer сидит между приложением и PostgreSQL и уменьшает число реальных соединений к базе. ()

7. Авторизация

Для обычного веб-приложения:

access token — короткий срок жизни
refresh token — дольше
refresh token хранить в httpOnly cookie
пароли хэшировать через argon2 или bcrypt

На backend:

JWT Guard
Role Guard
Rate Limit на login
2FA позже, если нужно

Особенно важно проверять права доступа на каждый объект. OWASP называет broken object level authorization одним из главных API-рисков: нельзя просто доверять userId или id, пришедшему с фронта. ()

9. Масштабирование

Для 20 000 зарегистрированных пользователей можно начать так:

1 frontend hosting
1 backend instance
1 PostgreSQL
1 Redis
1 worker

Потом масштабировать:

CDN
2–4 backend instances
Load Balancer
PostgreSQL managed instance
Redis managed instance
separate workers
monitoring

Express в своих production-рекомендациях тоже указывает на важность restart-механизма, кластеризации, кэширования, load balancer и reverse proxy для production-приложений. Даже если использовать NestJS, принципы те же, потому что это всё равно Node.js backend. ()

15. Главный совет

Для 20 000 пользователей тебе не нужна “архитектура как у Netflix”. Нужна нормальная, аккуратная база:

чистый React/Vite фронт
модульный backend
PostgreSQL
Redis
очереди
CDN
логирование
резервные копии
безопасная авторизация

Я бы начал именно с модульного монолита, потому что он проще, дешевле и быстрее. А когда появятся реальные нагрузки, отдельные части можно вынести: например, обработку картинок, уведомления, статистику или платежи.
